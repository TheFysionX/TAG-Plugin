# Security

## Request authentication

Pairing generates an Ed25519 keypair locally. Before network I/O, the connector persists the unpaired key and fixed exchange request body/request ID in `pending-device-secrets.json`, separate from any active device. On Windows it first replaces the connector-directory ACL with one inheritable current-user grant, then creates and explicitly verifies the pending file's user-only ACL before the code is sent. If the exchange response is lost after the short-lived code is consumed, the next `pair` resumes the same proof-of-possession request. The validated result writes config first and state last while retaining the pending recovery file, so either runtime-file write can be retried safely. During an intentional device replacement, the active key is not committed until the exchange succeeds and its final ACL is verified. The private key never leaves the machine. Authenticated responses return a request digest. The connector places it in the next request body and increments its sequence only after an accepted HTTP response. The backend can detect duplicate request IDs, stale timestamps, replayed sequences, gaps, rollback, and divergent digest chains.

Before transmission, the connector atomically stores a content-free pending request with its fixed request ID, sequence, canonical body, and HMAC-only cursor commit. If a response is lost or the process crashes before saving the receipt, the next run replays exactly that request. The backend returns the original idempotent result only when request ID, sequence, route, and body digest all match.

Journal collection covers all safely discoverable retained history and reduces it into content-free provider/source-session/UTC-hour/raw-model/raw-mode/surface aggregates, newest hour first. Collection is deliberately independent from the server's score-eligibility horizon; the server decides whether an otherwise valid raw event is eligible for Kleos. Stable session-hour aggregates are not sharded, so an aggregate above the server's single-event volume boundary remains intact for anomaly and quarantine handling. Catch-up is divided into pages of at most 5,000 events. Page zero is embedded in `state.json`; future pages are written with user-only permissions under `sync-pages/<batch-id>/` before state points to the batch. State holds a manifest with the expected index, event count, and digest for every page. Loading validates the bounded file size, path, schema, event allowlist, count, and digest; a missing or corrupt active/future page fails closed without committing collection cursors. Every ingest request contains either at most three events or at most two checkpoints, never both. Immediately before an eligible Codex checkpoint plan, a signed heartbeat returns the authoritative active snapshot or an explicit legacy/no-active null. Codex checkpoints form a parent-linked snapshot generation: only changed dates are sent as `daily_delta`, and the final lifetime `commit` marker carries the generation, parent, full merged-snapshot digest, and delta count. Every submitted checkpoint requires exactly one matching `submittedCheckpointCanonical: true` result; a commit also requires `submittedGenerationActive: true`. The connector's submitted generation, digest, lifetime value, and daily map advance only after that proof and every remaining outbox item succeed. A canonical commit rejected specifically for `stale_parent` hydrates the winner and retires only the stale checkpoint generation; already-collected private event pages remain durable and drain before their source cursors commit. Other inactive-marker or missing-delta failures remain blocked. The exact active chunk is crash-replayed, page transitions are persisted, and final cursors commit only after the final page completes; the private batch directory is then removed. Chunks are paced 550 ms apart to remain below the normal device request window. An interactive sync processes at most 100 ingest requests; an hourly scheduled run processes at most 1,000, inserts a signed heartbeat after each 50 ingest requests, and sends a final heartbeat even when a persisted backlog remains. Heartbeat and ingest share one monotonic sequence/digest chain, so neither can skip an unresolved pending request. Network failures, timeouts, HTTP 429, and HTTP 5xx responses use at most three attempts with exponential jitter or a bounded standard `Retry-After`. Permanent 4xx payload rejections are never blindly replayed forever: eligible invalid same-type chunks are bisected, while an isolated unacknowledged event or checkpoint remains visibly blocked and cannot advance its cursor. Every submitted event requires exactly one matching backend result with `submittedRevisionActive: true`, or `submittedObservationCanonical: true` when an exact content duplicate is canonical under another logical event. Aggregate counts and status labels are insufficient. Raw-only usage additionally requires `rawPreserved: true`; absent proof fails closed because no other record can preserve those tokens.

The v0.1.6-to-v0.1.7 bridge has no locally committed snapshot generation. Any already-persisted signed v0.1.6 outbox drains unchanged; the backend may retain its legacy checkpoints for audit but does not activate them into the generation projection. After signed hydration explicitly reports legacy/no modern head, the next fresh coherent snapshot deliberately uses the all-zero genesis parent and sends every daily value currently supplied by Codex. The backend may accept that genesis takeover exactly once only when its active checkpoint generation is explicitly legacy-prefixed; after takeover, ordinary parent equality is mandatory. A reinstall with a modern head hydrates that head and never blind-resets to genesis. The connector never discards pending signed legacy work or guesses a backend legacy generation ID from the retired v0.1.6 checkpoint hash.

Plan metadata is isolated from accounting. Codex plan/rate-window values come from official app-server account RPCs. Claude uses a currently successful first-party account status plus an exact allowlisted `rate_limit_tier` from account-bound provider process metadata or Claude Desktop's cached account bootstrap. The cache parser checks a cumulative-work-bounded set of recent values, uses the V8 structured-clone decoder, requires the exact account/membership/organization path, matches the current organization ID, requires the `claude_max` capability and an allowlisted billing type for a positive Max claim, and fails closed on malformed, oversized, stale, conflicting, wrong-account, or conversation-only data; it never opens Claude credentials or cookies. A temporary same-account exact-evidence miss emits no coarse Claude plan replacement. A same-account membership without `claude_max`, a successful logout, API-key/third-party auth route, or an authenticated organization switch emits an unverified replacement and closes the prior claim; the organization ID is converted to a local installation-keyed HMAC alias and never uploaded. Antigravity token parsing remains isolated from plan detection and never changes Antigravity settings. Its plan detector accepts only the version-pinned app, same-user Antigravity language-server executable, loopback listener, and pinned local TLS certificate; it treats `GetUserStatus.userTier` as the effective quota authority and `GetLoadCodeAssist.paidTier` only as corroboration. A disagreement or future unverified ID cannot be guessed into a retail plan. The verified `free-tier` means Antigravity Starter quota—Free or AI Plus—not exact Free. Missing or transient status preserves the last successful observation, while explicit logout closes it. Legacy `planInfo`, `teamsTier`, `pro`, `currentTier`, and `allowedTiers` fields are ignored. The CLI status-line fallback remains separately consented. The heartbeat carries only bounded allowlisted observation fields. A quota reset is accepted only when a stored window boundary advances, not when a percentage changes. DeepSeek `api_payg` requires explicit API evidence, so a hosted DeepSeek model cannot fabricate a subscription/API plan. Grok Build session summaries are not a token ledger and cannot create uploadable accounting usage.

Usage identity is split deliberately. `serviceProviderId` is the consented host service and is the only identity used for authorization, plan/reset lookup, account checkpoints, Raw totals, and provider filters. `provider` is the actual model vendor and is used only for registered-model validation and Kleos. The server rejects impossible host/surface combinations, so a foreign model hosted through an authorized service cannot escape that service's accounting boundary or manufacture another provider's plan.

Raw source event IDs are HMAC-SHA-256 derivations under a stable, server-issued per-account namespace and content-independent provider identities. Claude uses its message ID. Codex uses the first owner session identity plus lineage epoch and cumulative endpoint; copied or overlapping lineage is suppressed before identity is minted. Kimi prefers a bounded provider record ID when present and otherwise uses session/agent identity plus normalized timestamp occurrence ordinal. The aggregator counts each raw source ID once locally. Codex and Claude logical aggregates retain the released v4 identity domain while Kimi retains v3; both bind provider, stable source-session scope, UTC hour, raw source-model token, raw mode token, and coding surface under the same account namespace. Parser/accounting generation, collector provenance, source-ID set, accumulated totals, canonical registry mapping, and raw-versus-attributed projection do not participate in aggregate identity. A changed canonical observation is request-digested as another revision under the stable logical ID instead of creating separately scoreable usage. Codex, Claude, and Kimi each have an independent aggregate-through watermark. A provider advances only after its enabled source is discovered and collected completely; disabled, unavailable, or truncated providers retain their prior watermarks. Those staged watermarks and all file cursors commit only after the final sync page completes. Records without a valid timestamp are withheld from upload. Raw provider identities, source IDs, raw identity tokens, and the namespace key never upload as usage-event fields. The separate random installation HMAC remains limited to local path aliases and cursors, and identical records are not linkable across The Artificial Games accounts.

Codex and Kimi byte cursors persist a metadata journal identity, a complete-line boundary, and a rolling prefix digest over only allowlisted usage/model/mode fields. Before resuming, the parser rescans the bounded prior prefix and compares that safe digest. A changed allowlisted record anywhere before the cursor, an invalid boundary, or different file identity resets parsing to byte zero. Prompt, response, command, and tool content is neither stored nor hashed. Individual journal lines are capped at one MiB; an oversized line is discarded through its newline, counted as a parse loss, and does not grow memory without bound. Any malformed or oversized usage candidate makes that provider's coverage explicitly partial and keeps its repair status pending with only a count and timestamp; the completed range can still advance so normal new usage is not trapped behind permanently unreadable history. Unknown models and unclassified speed no longer wait in a lossy queue. They upload immediately with the atomic raw-only sentinel, retain the same stable source-derived event ID and exact token components, omit the source-model token, and are structurally ineligible for scoring. The v5 cursor generation performs one idempotent retained-history repair scan; existing scored IDs replay as duplicates, while usage withheld by the retired queue is recovered as raw-only.

All state-changing commands use one local lock with a random owner token. Its open file handle renews the filesystem lease every quarter of the 15-minute stale window. A stale contender atomically renames and revalidates the observed lease, refuses takeover while the recorded local PID is alive, and creates the replacement with exclusive-create semantics. Release checks both owner token and file identity, so a delayed old owner cannot delete a replacement lock.

Runtime JSON commits use an exclusive-create temp followed by one atomic rename, and ordinary failures remove the exact temp they created. Windows `EEXIST` or `EPERM` retries that same atomic primitive five times with bounded delay; exhaustion never unlinks the prior committed JSON. A hard process crash may bypass temp cleanup. After the same 15-minute safety window, a state-changing command that has proved ownership of the canonical overlap lock may delete only regular files under connector home whose name exactly matches the connector's runtime-JSON atomic format, whose encoded PID is no longer live, and whose unchanged canonical counterpart still exists as a regular file. A temp that is the sole recovery copy is preserved. Enumeration is bounded and limited to connector-home root plus direct files in exact sync-page batch directories; there is no recursive or wildcard deletion. Fresh files, live-writer files, malformed names, unrelated files, directories, and paths outside connector home are preserved.

## Supply chain

- Install the GitHub release asset whose tag resolves to the expected full pinned commit, never a moving branch.
- Compare the archive to the out-of-band expected SHA-256, then to its `SHA256SUMS` entry, and verify the GitHub artifact attestation.
- The package has no runtime or development dependencies. It includes the synthetic test suite and synthetic JSONL fixtures so `npm test` runs against the exact attested release artifact; no real user journal is packaged.
- `private: true` prevents accidental npm publication; official releases should be downloadable GitHub archives.
- Every third-party GitHub Action is pinned to a verified full commit SHA with its release version documented in a comment.
- Before packing, the release workflow rejects `-local`, requires the tag to equal `v` plus `package.json.version`, and requires package, runtime constant, and install-manifest versions to match. It then tests the source, creates a tarball, extracts and tests that exact package, publishes a checksum, and produces build provenance. It does not claim reproducible/deterministic archives until clean-build digest equality is separately proven.
- A confirmed install copies the verified release into a stable current-user version directory, writes a stable launcher and atomic current/previous pointer, and gives the scheduler the launcher plus an explicit connector-home argument. Existing `v0.1.9` installations need one explicit bootstrap install of the first updater-capable release.
- After a successfully committed signed heartbeat, the updater accepts only a newer stable offer with the exact official repository, version/tag, full commit, asset name, SHA-256, updater protocol, and runtime-state schema. It independently resolves the official GitHub tag and published non-draft/non-prerelease release, accepts only the exact asset through a bounded GitHub redirect chain, verifies its SHA-256, rejects unsafe archive paths/links/duplicates, and checks the dependency-free package contract before writing a new immutable version directory and atomically switching the pointer. It uses a separate update lock and privacy-safe update state. An update error is recorded safely and cannot roll back or invalidate the committed heartbeat, pairing, durable state, or pending outbox. This is GitHub metadata and checksum verification; it does not claim local artifact-attestation verification or an offline release-envelope signature.
- Never execute a remote script through a shell pipe.

## Local privilege

TAG Plugin needs only current-user read access to explicitly approved provider usage surfaces, write access to its own state directory, and outbound HTTPS to the configured origin for The Artificial Games. It does not need root/administrator rights, accessibility permissions, browser access, full-disk access, user source-repository access, or provider credentials. POSIX private files use mode `0600`; Windows pairing restricts the plugin directory before credential creation, verifies current-user-only ACLs on both pending and final secret files, and confirmed install re-verifies the final ACL before scheduler creation.

## Reporting a vulnerability

Use this repository's private vulnerability-reporting form under **Security → Advisories**. Repository maintainers must enable private vulnerability reporting before the first public release. Do not include real prompts, credentials, journal files, or private keys in a report; replace them with minimal synthetic reproductions.
